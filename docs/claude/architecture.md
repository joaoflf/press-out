# Architecture Details

## Request Flow

1. User uploads video via HTMX form -> `POST /lifts` handler
2. Handler saves file to `data/lifts/<id>/original.mp4`, creates DB row
3. Pipeline launches in a goroutine using **server context** (not request context)
4. Each stage runs sequentially; SSE broker publishes progress events per lift ID
5. Frontend receives SSE events via `GET /lifts/{id}/events` and swaps HTML with HTMX

## Pipeline Stage Interface

```go
type Stage interface {
    Name() string
    Run(ctx context.Context, input StageInput) (StageOutput, error)
}
```

- `StageInput`: LiftID, DataDir, VideoPath (cascading — each stage can update the path)
- `StageOutput`: VideoPath (next stage uses this as input)
- Unimplemented stages use `StubStage` (no-op, passes video through)
- Failed stages get `StateSkipped`; pipeline continues

## Stage Details

### Pose (stage 1)
- Runs `uv run scripts/pose.py` as subprocess
- Outputs `keypoints.json` with per-frame bounding boxes and 17 COCO keypoints
- Format: `{sourceWidth, sourceHeight, frames[]{timeOffsetMs, boundingBox, keypoints[]}}`

### Trim (stage 2)
- Uses keypoint displacement from `keypoints.json` to detect lift start/end
- Outputs `trimmed.mp4` via FFmpeg

### Crop (stage 3)
- Uses per-frame `boundingBox` from `keypoints.json` for crop region
- 9:16 aspect ratio, 5% padding
- Outputs `cropped.mp4` and `crop-params.json` (coordinate transform for skeleton stage)

### Skeleton, Metrics, Coaching (stages 4-6)
- Currently stub implementations

## SSE Architecture

`sse.Broker` manages per-lift subscriber channels. Pipeline publishes two event types:
- `pipeline-stages`: full checklist HTML for detail page
- `pipeline-status`: compact badge HTML for list page
- `pipeline-done`: signals completion

## Template System

Go `html/template` with clone pattern:
- Base layout parsed from `web/templates/layouts/`
- Partials parsed from `web/templates/partials/`
- Each page clones base + parses its own file from `web/templates/pages/`
- Template map keyed by filename (e.g., `tmplMap["lift.html"]`)

## Database

SQLite via `mattn/go-sqlite3` (requires CGO). Migrations are plain SQL files in `sql/schema/` applied in order at startup. Queries generated by sqlc from `sql/queries/`.

## E2E Tests

Use `chromedp` (headless Chrome). Tests in `test/e2e/` spin up a full test server with real DB, then drive the browser. Each test creates its own temp data directory and DB.
